旷厂三个月

发表于 2019-06-26 | 分类于 实习

3 月 25 入职, 6 月 26 正式离职，刚好在旷视实习三个月。记一下第一次实习的经历。

旷厂的氛围总体来说是轻松愉快，部门的人都很好，特别是靖入大哥给了我很多帮助，跟着他也学了很多东西。第一周就是配环境。配环境基本就是配置容器环境。第一次接触 docker 和 kubernetes，摸索了很久。

之后就是学习 kubernetes 组件，重点是 scheduler。还有学习 GO 的基本用法。之后工作就慢慢进入正轨，可以接一些需求了。

云平台

部门主要是做云计算平台，这是我第一次认识云计算。

云服务分为三大类: 基础设施即服务(IaaS)，软件即服务(SaaS)，平台即服务(PaaS)

云计算平台就是将开发者需要的软件集成到一个平台上，开发者登录这个平台，就可以使用平台上的开发环境，软件，数据库。

一个云平台总体需要以下服务

  • 前端显示界面
  • 用户登录认证
  • 存储管理
  • 分配 IP
  • 机器上部署的服务
  • scheduler
  • 物理机维护
  • ssh 代理
  • http 代理
  • 公网服务

网络服务不是很清楚，主要是网络隔离，和服务代理，负载均衡。

资源利用率

资源共享的过程离不开两个维度：资源粒度、时间片。资源的效率 =Σ(在线资源粒度时间片利用率) -Σ(资源离线粒度 * 时间片)。资源效率包括两部分，在线部分和离线部分，离线部分主要是机器故障到恢复的过程，优化这部分资源效率，提升服务器在线率。软硬件故障通用措施有：故障预警、故障检测、故障自动修复、故障自动化处理流程等。

  • 一定不能被分配的资源。比如：计算节点上没有 CPU，那么可以认为其他所有剩余的资源都是碎片

  • 不容易被分配的资源。比如：计算节点上剩余 CPU 或 Mem 较少，那么 GPU 能被分配出去的可能性比较低；剩余的资源比较小，只能满足小资源请求

  • 考虑节点上每种资源的剩余比例，找出比例最小的一个作为该节点的可用资源比例，该节点上所有资源都以该值作为计算依据。

  • 可以考虑以上面的值为依据做加权平均，上面的值越小，在碎片化率中的权重越高

  • 更重要的资源碎片化权重更高，比如给与 GPU 碎片更高的权重

配额管理

配额管理作为逻辑概念，不予实际物理资源绑定，作为调度前的检查。
凡是资源都可以设置配额，CPU，GPU，标签等

配额管理可以分为

  • 固定配额: 实例启动前和运行期间，实例占用的资源规格保持不变。或者物理机或者集群在混部过程，对某一类型的任务分配固定的资源占比。
  • 动态配额: 是指实例运行期间，CPU 或 Memory 等根据实时运行需求，进行动态的调配，只要所在物理机资源够用。动态配额模型，一旦支持动态的增加，也就意味着支持动态的减少，也就是一旦资源不够用的时候，高优先级任务继续运行，低优先级任务逼迫释放资源，甚至 Kill 进程或者服务。

调度流程

  1. 配额检查
  2. 初选：根据标签和剩余资源选出符合要求的计算节点
  3. 优选: 根据标签，剩余资源，拓扑状态为每个计算节点打分，选出得分最高的节点作为结果
  4. 绑定计算节点
  5. 启动任务

本质上是贪心策略，调度结果不一定是最优解

全局最优解: 参考 Firmament 的实现进行改进。
